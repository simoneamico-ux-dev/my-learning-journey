Introduction to Git

Git is like a "magic diary" for your code. Imagine writing a book: every time you finish a chapter, you take a photo of the page. If you mess something up later, you can always go back to that photo!

Git keeps track of all the changes you make to your files. It's like having a time machine for your project: you can see what you changed, when you changed it, and why you changed it.

git --version
  • Keeps track of modifications
  • Allows you to go back in time
  • Facilitates teamwork

Basic Git Commands

Basic Git commands are like the most important words of a foreign language. With these few commands you can already do a lot!

The first command you'll always use is git init, which transforms a normal folder into a "Git warehouse". It's like saying "from today this folder is special, keep track of everything!"

Then there's git add, which is like putting files in a box before shipping it. And git commit is like sealing the box and writing on it what it contains.

git init git add . git commit -m "My first commit"
  • git init: initializes a repository
  • git add: prepares files for commit
  • git commit: saves changes permanently

Understanding GitHub

GitHub is like a "social network for programmers". It's a place where you can store your Git projects online and share them with the world.

Think of it as Instagram, but instead of photos, you share code. You can follow other programmers, "like" (star) their projects, and even contribute to them!

GitHub makes collaboration incredibly easy. It's like having a shared workspace where everyone can see what others are doing and contribute their ideas.

git clone https://github.com/username/repository.git git remote add origin https://github.com/username/repository.git
  • Repository hosting: store projects online
  • Collaboration tools: work with others
  • Open source community: share and contribute

Working with Branches

Branches are one of Git's most powerful features. Imagine your project as a tree: the main trunk is the "main" branch, and you can create branches to try new features.

If you're working on a new feature, you create a separate branch. It's like having a copy of your project where you can experiment without breaking the original!

When your feature is ready, you can "merge" the branch with the main one. It's like grafting the branch onto the main tree.

git branch new-feature git checkout new-feature git switch -c another-feature
  • main/master: main branch
  • feature branches: for new functionalities
  • hotfix branches: for urgent bug fixes

Collaboration and Pull Requests

Pull Requests are the heart of collaboration on GitHub. It's like knocking on the door and saying "Hey, I improved your project, want to see?"

When you make a Pull Request, you're asking the project owner to "pull" your changes into their repository. It's a polite way to propose changes.

Before your changes get accepted, other developers can review your code, suggest improvements, or approve the changes. It's like having a group of friends check your homework before you turn it in!

git push origin my-feature
  • Fork: copy a repository to your account
  • Pull Request: propose changes
  • Code Review: other developers check your code

Git Configuration

Before using Git for the first time, it's like introducing yourself: you need to say who you are! Git needs to know your name and email to "sign" your commits.

Configuration is done once on your computer, and it's like filling out a digital ID card. Every time you make a commit, Git will use this information.

You can also configure your preferred text editor and many other details to customize Git according to your preferences.

git config --global user.name "Your Name" git config --global user.email "your@email.com" git config --global core.editor "code --wait"
  • Global configuration: applies to all projects
  • Local configuration: only for one project

Remote Repositories

Remote repositories are like "backup copies" of your project stored on the internet. The most famous is GitHub, but there are others like GitLab and Bitbucket.

It's like having your personal diary (local repository) and a copy in a bank vault (remote repository). If your computer breaks, the copy is always safe!

You can connect your local project to multiple remote repositories simultaneously. It's useful when you want to backup on different services.

git remote -v git push origin main git pull origin main
  • origin: standard name for the main repository
  • upstream: original repository when you fork

Merge Conflicts

Merge conflicts are like when two people modify the same sentence of a document simultaneously. Git doesn't know which version to keep!

Don't be scared of conflicts: they're completely normal when working in teams. It's like two chefs modifying the same recipe - someone has to decide which ingredient to use.

When it happens, Git shows you exactly where the differences are and you have to choose what to keep. It's like being a judge deciding between two versions.

<<<<<<< HEAD My version of the code ======= Someone else's version >>>>>>> branch-name
  • HEAD: the version in your current branch
  • Markers: <<<<<<< ======= >>>>>>> show the conflict

Git History and Logs

Git history is like a detailed log of everything that happened to your project. Each commit is like a diary entry with date, time, and description.

With git log you can see who did what and when. It's incredibly useful for understanding how the project evolved or finding when a bug was introduced.

You can also see differences between different versions, like comparing two photos of the same place taken at different times.

git log --oneline git show commit-hash git diff HEAD~1
  • git log: shows complete history
  • git show: details of a specific commit

GitHub Issues and Projects

GitHub Issues are like a public "to-do list" for your project. You can report bugs, propose new features, or ask questions.

It's like having a bulletin board where anyone can stick post-it notes with suggestions or problem reports. Much more organized than scattered comments!

Projects are like digital Kanban boards: you can organize Issues in columns like "To Do", "In Progress", "Done". Perfect for managing complex projects!

# In a commit message git commit -m "Fix login bug - closes #42"
  • Bug reports: error notifications
  • Feature requests: proposals for new functions

Git Best Practices

Using Git well is like driving: there are unwritten rules that make everything safer and more organized for everyone.

Commit messages should be clear and descriptive, like photo captions: "Add login function" is much better than "random stuff".

Make small and frequent commits instead of one giant one at the end. It's like saving a document every paragraph instead of only at the end - if something goes wrong, you've lost less work!

# Good example git commit -m "Add user authentication system" # Bad example git commit -m "stuff"
  • Frequent and small commits
  • Descriptive messages
  • Branch for each feature

Troubleshooting Common Issues

Everyone makes mistakes with Git, even experienced programmers! It's normal and there are always solutions. It's like learning to ride a bike - some falls are inevitable.

The most useful command ever is git status. It's like asking "where am I?" when you're lost. It always tells you what's happening and what you can do.

If you made a wrong commit, don't panic! Git keeps track of everything and you can almost always go back. It's like having a very powerful "undo" button.

git status git reset --soft HEAD~1 git stash
  • git status: to always orient yourself
  • git reset: to undo commits
  • git stash: to save temporary work
